# Socket Room AI And Realtime Lifecycle

- This file documents the room AI behavior that lives directly inside the Socket.IO server in backend/index.js.
- The room AI path is the most stateful AI path in the backend because it combines membership, live events, quota, persistence, and insight refresh in one flow.
- Use this file when modifying room presence, @ai behavior, or live room message handling.

## Socket Authentication And State

- Every socket is authenticated through backend/middleware/socketAuth.js before it can emit room events.
- The authenticated socket receives socket.user with at least an id and username.
- The runtime tracks per-room users in roomUsers.
- The runtime tracks global online users in globalOnlineUsers.
- The runtime tracks per-room typing state in typingUsers.
- The runtime tracks anti-flood counters in socketFlood.
- Flood control allows 30 events per 10-second window by default before the server returns an error payload.

## Client To Server Event Map

- authenticate acknowledges the authenticated user payload.
- join_room ensures the room exists and the user can be treated as a member.
- leave_room removes the socket from the room and updates presence.
- typing_start broadcasts a typing indicator with auto-expire behavior.
- typing_stop clears typing state explicitly.
- mark_read transitions message statuses forward only from sent or delivered to read.
- send_message persists non-AI room messages and supports file attachments.
- reply_message persists room replies with a reply preview object.
- add_reaction toggles supported emoji reactions in place.
- trigger_ai runs the room AI generation flow.
- edit_message edits an existing user-owned non-AI message inside the allowed time window.
- delete_message soft-deletes a message and refreshes room insight.
- pin_message and unpin_message manage moderator-only room pin state.

## Join And Presence Rules

- The socket join_room handler validates the room id and loads the room first.
- If the user is not already a room member, the handler can auto-add the user when capacity allows.
- Joining a new room leaves previously joined rooms for the same socket session.
- Presence broadcasts include user_joined, user_left, and room_users payloads.
- Global status broadcasts include user_status_change events on connect and disconnect.
- The runtime also updates User.onlineStatus and lastSeen in MongoDB.

## Plain Room Message Lifecycle

- The server normalizes content into trimmed textContent.
- The server validates attachment metadata through validateAttachmentPayload.
- The handler rejects empty messages that also lack a file.
- The handler rejects messages over 4000 characters.
- The handler requires both room membership and an active socket-room join.
- The server persists a Message document with isAI false.
- The handler may mark the new message as delivered immediately if another room member is online.
- The handler refreshes room insight after saving.
- The handler emits receive_message with the formatted message payload.
- Ordinary room messages do not extract memory entries in the current implementation.

## Room AI Lifecycle

- The frontend sends a room message first when the user types @ai, then separately emits trigger_ai with the stripped AI prompt.
- The server resolves requestedModel up front through resolveModel(modelId) for logging and error reporting.
- The server rejects empty prompts unless an attachment is present.
- The server rejects prompts over 4000 characters.
- The server validates attachment metadata before the AI call starts.
- The server emits ai_thinking with roomId and status true before running the request.
- The server checks in-memory AI quota through consumeAiQuota keyed by user:<id>.
- The server revalidates room membership and active room join state before generating.
- The server retrieves up to five relevant memories for the triggering user.
- The server loads or generates the room insight so the prompt can include a current structured summary.
- The server upserts new memory candidates from the AI prompt itself with sourceType room.
- The server logs the trigger with the requested or resolved model and provider.
- The server calls sendGroupMessage with room history, prompt, username, memory entries, insight, room name, model id, and attachment.
- On success, the server appends a user turn and a model turn to Room.aiHistory.
- The server trims Room.aiHistory to the first two seed items plus the latest 38 items when the history grows too large.
- The server persists a room Message document for the AI reply with userId ai and isAI true.
- The AI Message stores triggeredBy, modelId, provider, and top memory refs with summary and score.
- The server marks retrieved memories as used after persisting the AI response.
- The server refreshes room insight after persisting the AI response.
- The server emits ai_thinking with roomId and status false when the request completes.
- The server emits ai_response with the formatted AI message payload.
- The server returns an ack payload with success true and the message payload.

## Room AI Failure Lifecycle

- Gateway errors are normalized enough that the socket layer can still report modelId and provider to the client when possible.
- On failure, the server logs the failed model and provider pair.
- On failure, the server always clears the ai_thinking flag.
- On failure, the server persists an AI-authored room message that apologizes for the request failure.
- On failure, the server emits that fallback error message through ai_response so the room sees an explicit result.
- On failure, the ack payload returns success false, error AI request failed, and optional model and provider metadata.

## Server To Client Event Notes

- receive_message carries normal message payloads with attachment fields and message status.
- ai_response carries AI-specific payloads including isAI, triggeredBy, memoryRefs, modelId, and provider.
- reaction_update carries the updated reaction map for one message.
- message_status_update carries sent, delivered, or read transitions.
- message_edited, message_deleted, message_pinned, and message_unpinned mirror moderation and editing state changes.
- typing_start and typing_stop are broadcast to peers, not echoed back to the same author as a UI concern.
- error_message is the generic event used by emitSocketError for room-level failures.
