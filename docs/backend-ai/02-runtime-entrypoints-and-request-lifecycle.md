# Runtime Entrypoints And Request Lifecycle

- This file explains how the backend boots and how AI requests move through middleware, services, persistence, and real-time emission paths.
- The most important file in this topic is backend/index.js because it owns both app startup and room AI socket behavior.
- Use this document when you need to understand where to attach new middleware, where to add observability, or how the REST and socket paths differ.
- The emphasis here is lifecycle order, not just route names.

## Boot Sequence

- The backend loads environment variables from backend/.env through dotenv in backend/index.js.
- The runtime creates an Express app and an HTTP server, then attaches a Socket.IO server to the same HTTP server instance.
- Passport configuration is loaded before route mounting so auth flows are ready for both local and Google login routes.
- CORS uses CLIENT_URL or falls back to http://localhost:5173.
- JSON request bodies are limited to 5mb at the Express layer.
- Request logging middleware stamps req.requestId, adds X-Request-Id, and logs start and finish events.
- The general API rate limiter is mounted on /api before specific routes are mounted.
- All REST route modules are mounted before the /api/health endpoint is defined.
- Socket.IO is created after the Express app and uses the same CORS origin as the REST app.
- Socket authentication is applied through io.use(socketAuthMiddleware) before any room event handlers run.
- At startup, the backend connects to MongoDB through connectDB().
- After the database connects, the runtime calls refreshModelCatalogs() and logs configured provider counts.
- Only after database connection and initial model refresh does the server listen on PORT.

### `backend/index.js`

- Responsibility: Owns startup, request logging, socket state, and room AI execution.
- Mounts AI-related REST routes before the health check.
- Creates shared in-memory maps for room presence, online users, typing state, and flood control.
- Formats outbound room messages with memoryRefs, modelId, and provider when present.
- Calls refreshModelCatalogs() at startup so model lists are warm before the frontend fetches them.
- Runs trigger_ai in the socket layer rather than delegating to a separate route module.
- Refreshes room insight after message sends, replies, AI responses, edits, deletes, and pin state changes.

### `backend/middleware/rateLimit.js`

- Responsibility: Defines auth, AI-route, and general API throttles.
- The AI route limiter uses a 15-minute window with a default cap of 80 requests.
- The general API limiter defaults to 1000 requests per 15 minutes.
- Health checks and auth routes skip the general limiter.
- Limiter keys prefer authenticated user ids when available.
- Limiter responses include retryAfterMs and requestId when possible.

### `backend/middleware/aiQuota.js`

- Responsibility: Adds a second AI quota layer on top of the Express rate limiter.
- The middleware keys on user:<id> when authenticated and ip:<ip> otherwise.
- The middleware returns 429 with retryAfterMs if the quota is exhausted.
- This quota is service-backed and intentionally separate from the route limiter.
- The middleware stores the quota result on res.locals.aiQuota for downstream use if needed.

### `backend/services/aiQuota.js`

- Responsibility: Tracks AI quota usage in memory.
- The default quota window is 15 minutes.
- The default quota cap is 20 requests per key per window.
- Quota resets when the process restarts because state is stored in a local Map.
- The service returns allowed, remaining, and retryAfterMs rather than throwing.

## Solo REST Lifecycle

- A solo request enters through POST /api/chat in backend/routes/chat.js.
- The auth middleware resolves the user before the chat handler runs.
- The AI quota middleware runs before the main handler and can stop the request with 429.
- The chat route validates message presence and attachment metadata first.
- If a conversationId is provided, the route loads the conversation for the current user.
- If a projectId is provided, the route loads the project and rejects cross-user or mismatched project usage.
- The route retrieves relevant memories through retrieveRelevantMemories before generation.
- If the conversation already exists, the route attempts to fetch a cached insight through getConversationInsight.
- The route calls sendMessage in the AI gateway with history, memory entries, optional insight, model id, attachment, and project context.
- The AI gateway returns content plus selected model, provider, usage, processing time, and routing metadata.
- The chat route persists both the new user turn and the assistant turn to Conversation.messages.
- The assistant turn stores memoryRefs, modelId, provider, requestedModelId, token fields, autoMode, autoComplexity, and fallbackUsed.
- The route upserts new memory candidates from the user prompt and marks retrieved memories as used.
- The route tries to refresh the conversation insight after persistence, but a refresh failure only logs a warning and does not fail the chat response.
- The final HTTP response returns the assistant content plus metadata the frontend uses for display and diagnostics.

## Room Socket Lifecycle

- A room session starts when the authenticated socket connects and exposes socket.user through the socket auth middleware.
- The frontend normally fetches room detail over REST first, then emits join_room to establish live room presence.
- The join_room handler can auto-add a user to the room if the user is not already a member and the room has capacity.
- Once joined, the socket server tracks room presence in memory and broadcasts room_users.
- Plain room messages go through send_message, which persists the message and refreshes room insight.
- Replies go through reply_message, which validates the parent message and blocking rules before saving.
- Room AI goes through trigger_ai, not through send_message itself.
- The trigger_ai handler validates prompt length, attachment metadata, quota, membership, and socket room membership.
- The server emits ai_thinking before generation and clears it on success or failure.
- Before generation, the handler loads relevant user memories and the current room insight.
- The handler also upserts memory candidates from the AI prompt itself.
- The handler calls sendGroupMessage in the AI gateway with room history, prompt text, username, optional insight, memory entries, attachment, and model id.
- On success, the handler appends both user and model turns to Room.aiHistory and trims the history window.
- The handler then persists a room Message document for the AI response with userId: ai and isAI: true.
- The server marks used memories, refreshes room insight, emits ai_response, and returns an ack payload.
- On failure, the server persists a fallback AI error message to the room and emits both ai_response and an ack error payload.

## Upload Lifecycle

- Files are uploaded through POST /api/uploads before the frontend references them in chat or room AI payloads.
- The upload middleware writes the file to backend/uploads with a random 16-byte hex filename plus an allowed extension.
- Upload responses return fileUrl, fileName, fileType, and fileSize.
- Attachment validation is centralized in backend/services/messageFormatting.js so both solo and room flows use the same rules.
- Text-like files can be read directly into prompts by the AI gateway.
- Small images can be converted to data URLs and passed to providers that support image inputs.
- PDFs currently contribute metadata only because PDF text extraction is not enabled.

## Lifecycle Risks And Extension Rules

- If you add a new AI route, decide explicitly whether it needs both the Express AI rate limiter and the in-memory AI quota middleware.
- If you add a new room AI behavior, keep the ack contract compatible with the frontend emitWithAck helper in frontend/src/hooks/useSocket.ts.
- If you change attachment rules, update upload acceptance, shared payload validation, and prompt assembly together.
- If you introduce distributed deployment, replace in-memory quota and presence state with a shared backing store.
- If you add new prompt templates, surface them through the admin prompt routes only when the runtime can actually consume them.
- If you move room AI out of backend/index.js, preserve the event names and room insight refresh behavior to avoid frontend regressions.
