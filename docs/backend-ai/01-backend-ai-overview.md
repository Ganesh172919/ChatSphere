# Backend AI Overview

- This document is the backend-only map of the current AI implementation as of March 31, 2026.
- The goal is to describe where AI behavior lives, what changed recently, and how to reason about the backend without reading every file from scratch.
- The current backend AI stack is multi-provider and no longer limited to a single direct Gemini-only path.
- Solo AI flows are REST-driven, room AI flows are Socket.IO-driven, and both persist model and provider metadata.
- Memory extraction, conversation insights, prompt templates, model discovery, uploads, and quota enforcement are all part of the same backend AI story.
- Use this file first, then continue through the remaining nine backend AI reference files for topic-level depth.

## Current Snapshot

- The backend starts in backend/index.js and mounts AI-related REST routes under /api/ai, /api/chat, /api/conversations, /api/memory, /api/settings, /api/admin, /api/export, /api/projects, and /api/rooms.
- The provider-aware gateway is backend/services/gemini.js, even though the filename still references the project earlier Gemini-first implementation.
- The gateway currently supports OpenRouter, Gemini direct, xAI Grok direct, Groq direct, Together AI, and Hugging Face router-based requests when matching API keys exist.
- The backend exposes a user-facing auto selection mode through /api/ai/models, then resolves that request to a concrete model before generation.
- Solo chat persists to Conversation.messages, while room AI responses persist to Message documents with isAI, modelId, provider, and memoryRefs.
- AI-derived structured data is persisted through MemoryEntry, ConversationInsight, and PromptTemplate documents.
- Feature-level enablement is stored on User.settings.aiFeatures and enforced inside utility routes such as smart replies, sentiment, and grammar.
- Admin users can update prompt templates through /api/admin/prompts/:key, which changes future AI behavior without code edits.
- Import and export flows are AI-aware because imported histories can generate memories and insights, and exported histories can include them.
- Uploads are validated once and reused across solo chat, room AI, and project context attachment flows.

## Latest Documentation-Relevant Changes

- The docs now need to reflect provider-aware routing instead of describing the backend as Gemini-only.
- The docs now need to reflect auto routing and model fallback instead of implying every request uses the explicit requested model unchanged.
- The docs now need to reflect the persistence of requestedModelId, processingMs, token usage fields, autoMode, autoComplexity, and fallbackUsed on solo assistant messages.
- The docs now need to reflect that room AI requests happen over the trigger_ai socket event and not through a room REST endpoint.
- The docs now need to reflect room insight refreshes after normal room messages, replies, AI responses, edits, deletes, and pin changes.
- The docs now need to reflect that ordinary room messages do not create memory entries; memory extraction is tied to AI-triggered room prompts.
- The docs now need to reflect that project context can be injected into solo AI prompts through projectId and Project.files.
- The docs now need to reflect that imports can generate both memory entries and conversation insights through shared services.
- The docs now need to reflect that there is currently no checked-in backend/.env.example, so setup instructions must be manual.
- The docs now need to reflect the exact analytics endpoints in code rather than older placeholder route names.

## Backend AI Flows At A Glance

- Solo AI flow: frontend/src/hooks/useChat.ts calls frontend/src/api/chat.ts, which calls POST /api/chat, which calls sendMessage, which calls the gateway, which persists the assistant turn and refreshes insight.
- Room AI flow: frontend/src/hooks/useSocket.ts emits trigger_ai, backend/index.js validates membership and quota, the gateway runs sendGroupMessage, and the AI reply is emitted as ai_response.
- Model discovery flow: frontend/src/api/ai.ts calls GET /api/ai/models, which refreshes provider catalogs and returns the visible models for the deployment.
- Smart reply flow: the frontend asks /api/ai/smart-replies, backend/routes/ai.js applies settings and fallbacks, and the response returns exactly three strings.
- Sentiment flow: the frontend asks /api/ai/sentiment, backend settings are checked, and a structured sentiment payload is returned for room message badges.
- Grammar flow: the frontend asks /api/ai/grammar, backend settings are checked, and the response returns corrected text plus optional suggestions.
- Memory retrieval flow: backend/routes/chat.js and backend/index.js both call retrieveRelevantMemories before generation when context should help the model.
- Memory persistence flow: upsertMemoryEntries writes or merges stable facts after solo user prompts, room AI prompts, and imports.
- Insight flow: conversation and room action endpoints call refreshConversationInsight or refreshRoomInsight to regenerate structured summaries on demand.
- Prompt override flow: prompt templates are resolved by key through getPromptTemplate, with database overrides taking precedence over hardcoded defaults.
