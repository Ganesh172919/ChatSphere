# Conversation Insights Summaries And Actions

- This file explains the structured insight system used for solo conversations and group rooms.
- Insights are not free-form summaries only; they persist title, summary, intent, topics, decisions, and action items.
- Use this file when changing summarization prompts, room action buttons, or the persistence model for structured AI outputs.

### `backend/services/conversationInsights.js`

- Responsibility: Generates, stores, refreshes, and returns structured insights.
- Builds a scope key from scopeType, scopeId, and optional userId.
- Creates a deterministic fallback summary when the model cannot return valid JSON.
- Generates insight prompts from the latest message window, not the entire historical corpus.
- Uses getJsonFromModel so insight generation can benefit from gateway fallback behavior.
- Persists insights through ConversationInsight.findOneAndUpdate with upsert true.
- Exposes dedicated refresh and read helpers for conversations and rooms.

### `backend/models/ConversationInsight.js`

- Responsibility: Stores structured AI summaries for conversation and room scopes.
- scopeType is either conversation or room.
- scopeKey is unique and indexed so repeated refreshes overwrite the same logical record.
- The payload stores title, summary, intent, topics, decisions, actionItems, and messageCount.
- The model currently stores promptVersion, but the service does not actively update it yet.

### `backend/routes/conversations.js`

- Responsibility: Makes conversation insights visible to the frontend.
- Conversation detail routes do not embed the insight by default; the frontend loads it separately or receives it from chat responses.
- Action routes regenerate insight on demand and return specialized slices such as summary, decisions, or tasks.
- Conversation action routes are owned by the authenticated user only.

### `backend/routes/rooms.js`

- Responsibility: Makes room insights visible to the frontend.
- Room detail routes embed the current insight for convenience.
- Room action routes regenerate insight on demand for summary, decisions, or action items.
- Room insight routes require room membership and do not expose insight to non-members.

## Insight Sources

- Conversation insight is generated from Conversation.messages.
- Room insight is generated from the latest room Message documents sorted by recency and then reversed into chronological order.
- Fallback titles come from conversation titles or the constant Room insight.
- Usernames are preserved in room serialization so the model can reason about who said what.

## Insight Refresh Triggers

- Solo chat refreshes conversation insight after a successful assistant response is persisted.
- Conversation action buttons refresh insight explicitly on demand.
- Room send message refreshes room insight.
- Room reply message refreshes room insight.
- Room AI response refreshes room insight.
- Room edit message refreshes room insight.
- Room delete message refreshes room insight.
- Room pin and unpin flows refresh room insight.

## Frontend Uses Of Insight

- Solo chat renders the insight in frontend/src/components/ConversationInsightsPanel.tsx.
- Group chat renders room insight in the same component with a room-specific heading.
- The frontend exposes three actions: summarize, extract tasks, and extract decisions.
- The frontend can optionally pass a modelId to insight endpoints so the user can align summary generation with the currently selected model.
- Insight data is treated as read-mostly UI state and refreshed on demand rather than continuously streamed.

## Fallback Behavior

- If structured JSON generation fails, the service builds a fallback summary from message text and tokenized topics.
- Fallback intent is question-answering when the text includes a question mark and discussion otherwise.
- Fallback decisions and action items are empty arrays rather than guesses.
- This fallback keeps the insight UI usable even when providers are degraded.

## Design Constraints

- Insight generation is deliberately concise: title length, summary length, topic count, decision count, and action item count are all capped.
- Room insight and conversation insight share one service so prompt changes can affect both surfaces.
- Insight generation is best-effort and should not be allowed to block the primary chat path.
- Structured fields are easier to filter, render, and export than one large summary blob.
