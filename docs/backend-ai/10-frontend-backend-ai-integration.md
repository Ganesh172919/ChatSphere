# Frontend Backend AI Integration

- This file explains exactly how the frontend and backend connect around AI features while keeping the implementation backend-centered.
- The aim is to make handoffs between frontend and backend work predictable: models, message shapes, settings, insights, attachments, and socket events all need to line up.
- Use this file when implementing a new frontend AI affordance or when diagnosing a mismatch between UI state and backend behavior.

### `frontend/src/api/ai.ts`

- Responsibility: Frontend REST wrapper for AI utility routes and model discovery.
- Calls /ai/models, /ai/smart-replies, /ai/sentiment, and /ai/grammar.
- Caches model discovery for 30 seconds to avoid unnecessary catalog refresh traffic.
- Defines the frontend AIModel shape as id, label, provider, and supportsFiles.

### `frontend/src/api/chat.ts`

- Responsibility: Frontend REST wrapper for solo AI chat.
- Calls POST /chat with message, history, conversationId, modelId, attachment, and projectId.
- Expects back the full assistant metadata used for UI explanation and conversation persistence.

### `frontend/src/hooks/useChat.ts`

- Responsibility: Frontend orchestration for solo chat.
- Builds local pending assistant messages before the backend returns.
- Passes stored conversation history to the backend in the parts-style format expected by the chat route.
- Updates local conversation state with modelId, provider, token counts, autoMode, autoComplexity, and fallbackUsed from the backend response.
- Fetches conversation detail and conversation insight separately when needed.

### `frontend/src/pages/SoloChat.tsx`

- Responsibility: Solo chat page that binds backend AI features to UI.
- Loads visible models through fetchAvailableModels().
- Persists the selected model in local storage under chatsphere.solo.model.
- Uploads files before calling the solo chat route so the backend receives a normalized attachment object.
- Shows grammar suggestions and smart replies using the same selected model when possible.
- Runs conversation insight actions and stores the returned insight in local state.

### `frontend/src/hooks/useSocket.ts`

- Responsibility: Frontend wrapper around room socket events.
- Authenticates the socket with the JWT access token.
- Reconnects with a refreshed token if the server rejects the current token.
- Wraps every client-to-server room event in emitWithAck so the backend ack contract becomes a Promise.
- Exposes triggerAi(roomId, prompt, modelId, attachment) as the frontend entry point to room AI.

### `frontend/src/pages/GroupChat.tsx`

- Responsibility: Room page that composes backend room REST and socket AI flows.
- Loads room detail through REST before joining the room socket.
- Loads room insight and room polls over REST.
- Loads the visible model catalog and stores the selected model under chatsphere.group.model.
- Uploads attachments first and then passes the normalized attachment object to either send_message or trigger_ai.
- Sends a normal room message and then conditionally emits trigger_ai when the input contains @ai.
- Shows ai_thinking, ai_response, sentiment badges, smart replies, grammar suggestions, and the room insight panel using backend data.

### `frontend/src/api/rooms.ts`

- Responsibility: Frontend room REST wrapper.
- Calls room detail, room insight, room action, join, and upload endpoints.
- Defines the GroupMessage shape with memoryRefs, modelId, and provider so room AI metadata renders correctly.

### `frontend/src/api/conversations.ts`

- Responsibility: Frontend conversation REST wrapper.
- Fetches conversation detail with persisted assistant metadata fields.
- Runs insight action routes for solo conversations.
- Keeps conversation detail aligned with what the chat route writes to Conversation.messages.

### `frontend/src/api/settings.ts`

- Responsibility: Frontend wrapper for user settings.
- Exposes aiFeatures.smartReplies, aiFeatures.sentimentAnalysis, and aiFeatures.grammarCheck to the UI.
- Keeps a short cache so repeated settings checks do not hammer the backend.

## Integration Contracts That Must Stay Aligned

- Attachment objects must always contain fileUrl, fileName, fileType, and fileSize.
- Solo assistant responses and conversation detail responses must expose the same metadata field names or the local conversation store will drift.
- Room AI message payloads from ai_response and room detail payloads from GET /api/rooms/:id should stay shape-compatible.
- The frontend expects the visible model list to group by provider and to include supportsFiles hints.
- The frontend expects auto to be accepted on requests but not necessarily persisted as the selected final model.
- The frontend uses retryAfterMs and human-readable error strings when available for user feedback.

## Current End To End Connection Patterns

- Solo AI: page to useChat to /api/chat to gateway to conversation persistence to optional insight refresh to UI update.
- Room AI: page to send_message socket to optional trigger_ai socket to gateway to message persistence to insight refresh to ai_response to UI update.
- Insight actions: page to room or conversation action REST route to refresh insight service to MongoDB upsert to UI update.
- Smart replies: component to /api/ai/smart-replies to gateway JSON generation or fallback to button rendering.
- Sentiment badges: page effect to /api/ai/sentiment to gateway JSON generation or fallback to badge rendering.
- Grammar suggestions: component to /api/ai/grammar to gateway JSON generation or fallback to suggestion bar rendering.

## Recommended Rules For Future Frontend Work

- Do not let the frontend guess the final provider or selected model; always trust backend response metadata.
- Do not bypass upload normalization by sending raw browser files directly to chat or socket AI routes.
- Do not assume room AI and solo AI share identical request contracts; room AI is socket-based and solo AI is REST-based.
- Do not assume a disabled feature only hides UI; the backend will reject the request explicitly.
- When adding a new AI feature, define the backend contract first, then add the frontend wrapper, then wire the page-level state.
- When adding a new metadata field, update backend persistence, backend read routes, frontend API types, and UI components in the same change.
