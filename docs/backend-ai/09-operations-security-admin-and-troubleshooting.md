# Operations Security Admin And Troubleshooting

- This file collects the operational concerns around the backend AI stack: rate limits, quotas, prompt administration, safety checks, and practical debugging guidance.
- Use this file when the AI system is failing, when you need to operate it in production, or when you are preparing to harden or scale the backend.

## Operational Guards

- AI utility REST routes use both aiLimiter and aiQuotaMiddleware.
- Solo chat uses aiQuotaMiddleware and relies on gateway error normalization for provider-specific failures.
- Room AI uses the in-memory consumeAiQuota service directly inside the socket handler.
- Room actions also pass through socket flood control before business logic begins.
- Attachment acceptance is restricted to explicit MIME types and a 5 MB maximum upload size.
- Attachment metadata is revalidated after upload before prompt assembly.

## Security-Relevant Behaviors

- REST and socket flows both require authenticated users before they can invoke AI features.
- Room AI and room insight routes both require room membership before data is exposed or prompts are built.
- Room reply and reaction handlers also check blocking relationships to prevent certain interactions between blocked users.
- Admin prompt routes require both auth middleware and the admin middleware.
- Uploads use randomized filenames and reject path traversal in the file serving route.
- Model provider API keys stay server-side and are never sent to the frontend.

### `backend/routes/admin.js`

- Responsibility: Administrative prompt and reporting surface.
- Prompt templates are the main admin-controlled AI configuration point in the codebase.
- Prompt writes are persisted to MongoDB and take effect without a restart because prompt resolution is dynamic.
- The same admin module also exposes user and report management, which matters operationally when AI moderation-adjacent workflows are reviewed.

### `backend/helpers/logger.js`

- Responsibility: Structured logging helper referenced by AI-related routes and gateway code.
- The gateway logs AI attempt, success, and failure events through the shared logger.
- Chat and utility routes log request ids so failures can be traced across middleware and provider calls.
- Insight refresh failures are logged as warnings instead of causing visible route failures in the happy path.

## Common Failure Modes

- No visible models in the frontend usually means no provider keys were configured or catalog fetches failed before a cached result was available.
- A 403 from smart replies, sentiment, or grammar usually means the user disabled the feature in settings.
- A 429 on utility routes can come from the Express AI limiter or the in-memory AI quota middleware.
- A 429 or 503 from solo chat can also originate from gateway-classified provider errors.
- Room AI errors can show up both as failed acks and as persisted apology messages in the room timeline.
- Room insight appearing stale often means a refresh did not run on the path you changed or a refresh failed and only logged a warning.
- Attachment failures often come from mismatched MIME types, missing fields, or oversized files.
- Imported histories that do not preview correctly usually need parser support in backend/services/importExport.js.

## Troubleshooting Checklist

- Check /api/health first to confirm the backend and database are up.
- Check startup logs for provider count output after refreshModelCatalogs().
- Check that the frontend received non-empty models from /api/ai/models.
- Check request ids in logs to connect route-level failures with gateway attempt logs.
- Check whether the user AI feature toggles are disabled in /api/settings.
- Check whether the process recently restarted, because quota and room presence state are stored in memory.
- Check uploaded file metadata in the browser payload and in the stored upload response.
- Check whether the requested model id still exists in the visible catalog or whether the frontend is holding a stale local-storage value.
- Check whether room users actually joined the socket room before expecting room AI to succeed.
- Check whether the error included retryAfterMs, modelId, or provider to distinguish provider throttling from local validation failures.

- Consider provider or feature-specific quotas when some tools are much cheaper or more expensive than others.
- Add provider health dashboards if model catalogs and generation quality become operationally critical.
- Add explicit PDF extraction support only after defining safety, size, and cost limits.
