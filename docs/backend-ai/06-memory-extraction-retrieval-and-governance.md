# Memory Extraction Retrieval And Governance

- This file explains how user memory works in the backend AI layer.
- The memory system is intentionally hybrid: deterministic rules extract obvious stable facts, and the model can add more candidates when helpful.
- Use this file when modifying long-term context behavior, memory ranking, manual editing rules, or import-derived memories.

### `backend/services/memory.js`

- Responsibility: Builds, stores, scores, retrieves, and marks memory usage.
- Normalizes text by lowercasing, stripping punctuation, and condensing whitespace.
- Tokenizes text into words longer than two characters for overlap scoring.
- Extracts deterministic memories such as name, location, employer, favorite items, and general likes from explicit phrasing.
- Calls getJsonFromModel to extract additional AI-derived stable memory candidates.
- Deduplicates memories using a SHA-1 fingerprint built from the normalized summary.
- Merges new candidates into existing entries by raising confidence and importance, resetting recency, and refreshing source references.
- Scores retrieved memories using text overlap, importance, confidence, recency, usage count, and a pinned bonus.
- Marks memory usage after successful AI generation by incrementing usageCount and setting lastUsedAt.

### `backend/models/MemoryEntry.js`

- Responsibility: Persists stable user memories and their scoring metadata.
- Stores summary, details, tags, and a unique fingerprint per user.
- Tracks sourceType, sourceConversationId, sourceRoomId, sourceMessageId, and sourceImportSessionId.
- Stores confidenceScore, importanceScore, recencyScore, pinned, usageCount, lastObservedAt, and lastUsedAt.
- Indexes by userId and fingerprint for dedupe and by recency-oriented fields for retrieval.

### `backend/routes/memory.js`

- Responsibility: Lets users inspect and curate memory entries.
- Search is substring-based over summary, details, and tags after the backend loads a capped memory set.
- Users can pin memories, which improves retrieval by adding a scoring bonus.
- Users can edit summary, details, tags, confidence, and importance when correcting model-generated memory.
- Users can delete memory entries that should not influence future prompts.
- Imports can preview candidate memories before they are committed.

## Where Memory Is Read

- Solo AI reads memory in backend/routes/chat.js before calling sendMessage.
- Room AI reads memory in the trigger_ai handler inside backend/index.js before calling sendGroupMessage.
- Memory retrieval is scoped to the authenticated user only.
- Memory retrieval is currently capped at five entries for active prompt injection in both solo and room AI flows.

## Where Memory Is Written

- Solo AI writes memory candidates from the user prompt after the assistant reply is persisted.
- Room AI writes memory candidates from the triggering AI prompt before generation completes.
- Import flows write memory candidates from imported user-authored text during the import commit stage.
- Manual memory editing routes let the user correct or remove stored memories after the fact.

## Scoring Model

- Text overlap contributes 45 percent of the final score.
- Importance contributes 20 percent of the final score.
- Confidence contributes 15 percent of the final score.
- Recency contributes 10 percent of the final score.
- Usage count contributes up to 10 percent through a capped bonus.
- Pinned memories receive an additional fixed bonus of 0.15.
- Entries with scores below the threshold are excluded unless pinned.

## Governance Rules

- Only memories that look stable and potentially useful later should be persisted.
- Temporary requests and generic chit-chat should be ignored by AI extraction prompts.
- Deterministic extraction intentionally prefers clear, explicit phrases over speculative inference.
- Pinned user-edited memories have special weight, so moderation or support tooling should be careful when bulk editing them.
- Fingerprint-based dedupe means summary wording changes can affect whether a new memory merges or creates a separate entry.
- Memory is user-scoped and not shared across users, even in group rooms.
- Room AI should use only the triggering user memory and not infer memory for the whole room.

## Operational Risks

- Inconsistent summaries can cause near-duplicate memories when the semantic fact is the same but the wording differs.
- Memory extraction quality depends on the structured JSON response from the gateway and can fall back to deterministic extraction only.
- Because scoring is local and heuristic, pinned and user-edited memories can outweigh fresher but weaker evidence.
- Memory growth is capped only indirectly by route behavior and retrieval limits; there is no dedicated archival policy yet.
- Imports can create many memories at once, so bulk-review UX matters for user trust.
